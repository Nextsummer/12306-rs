# 12306-rs 产品需求文档

## 1. 背景

用户在节假日或热门线路购票时，需要反复查询余票、筛选车次、选择乘客和席别，并在有票时尽快提交订单。手动操作成本高，容易错过余票。

`12306-rs` 旨在提供一个本机和 Docker 可运行的 Rust 服务，帮助用户完成 12306 车票查询、余票监控和自动提交订单。系统不自动支付，订单提交成功后提醒用户前往 12306 完成支付。

## 2. 产品目标

- 降低用户反复手动查票的成本。
- 让用户可以配置明确的抢票任务。
- 在符合条件时自动提交订单。
- 在无符合条件余票且用户开启强候补时自动提交候补订单。
- 同时支持 Web UI 和 CLI。
- 通过 SQLite 保存配置、任务、状态和日志。
- 支持 macOS、Linux 和 Docker 运行。

## 3. 用户角色

### 3.1 普通用户

需要查询车票、创建抢票任务，并在系统提交订单后完成支付。

### 3.2 高级用户

希望通过 CLI 管理任务，或在本机、服务器、Docker 中长期运行服务。

## 4. 使用场景

### 4.1 查询车票

用户输入出发站、到达站和日期，系统展示可选车次、席别、余票状态和无座状态。

### 4.2 创建普通抢票任务

用户选择日期、出发站、到达站、乘客、席别、车次偏好和轮询策略。系统持续查询余票，并在有符合条件余票时自动提交订单。

### 4.3 创建多日期任务

用户可以为同一出行计划选择多个日期。系统在多个日期中查询符合条件的车票，并按任务配置的优先级处理。

### 4.4 创建多乘客任务

用户可以选择多个乘客。系统提交订单时必须严格使用用户选择的乘客列表。

### 4.5 开启无座

用户可以把无座作为可接受席别。只有用户明确选择无座时，系统才会提交无座相关订单。

### 4.6 开启强候补

用户可以在任务中显式开启候补。开启后，当没有符合条件余票时，系统可以自动提交候补订单。强候补优先级高于继续刷票。

### 4.7 支付提醒

系统提交普通订单成功后进入待支付状态，提醒用户前往 12306 支付。系统不执行支付。

### 4.8 人工登录验证

用户使用账号密码登录。如果 12306 要求额外验证，系统在 Web UI 中引导用户人工完成验证。

## 5. 功能范围

### 5.1 登录

需求：

- 支持账号密码登录。
- 支持查看当前登录状态。
- 支持退出登录。
- 登录遇到额外验证时，Web UI 展示人工处理入口。
- CLI 登录遇到额外验证时，提示用户打开 Web UI 继续处理。

验收标准：

- 用户可以完成登录。
- 用户可以知道当前是否已登录。
- 登录失败时能看到明确原因。
- 不在普通日志中输出密码。

### 5.2 车票查询

需求：

- 支持输入出发站、到达站和日期查询。
- 支持展示车次、出发时间、到达时间、历时、席别、余票和无座信息。
- 支持按车次筛选。
- 支持按席别筛选。
- 支持无座显示。

验收标准：

- CLI 可以查询并输出结果。
- Web UI 可以查询并展示结果。
- 无票、余票充足、余票紧张、仅无座可用等状态可以区分。

### 5.3 乘客管理

需求：

- 支持读取或维护乘客信息。
- 支持任务选择多个乘客。
- 任务提交时必须使用用户选择的乘客。

验收标准：

- 创建任务时可以选择乘客。
- 多乘客任务可以保存和展示。
- 提交订单前可以确认乘客配置。

### 5.4 抢票任务

需求：

- 支持创建任务。
- 支持编辑未运行任务。
- 支持启动任务。
- 支持暂停任务。
- 支持恢复任务。
- 支持取消任务。
- 支持查看任务详情。
- 支持查看任务日志。
- 支持多任务同时运行。

任务配置项：

- 出发站。
- 到达站。
- 出行日期列表。
- 乘客列表。
- 席别列表。
- 是否接受无座。
- 车次偏好列表。
- 是否开启候补。
- 是否开启强候补。
- 查询间隔。默认 3000ms，最小 1000ms；不提供 500ms 这类高风险默认选项。
- 最早和最晚出发时间；最晚时间早于最早时间时表示跨午夜。
- 定时启动时间，保存到 SQLite。
- 高铁/动车座位位置偏好，每位乘客一个位置，可多选。
- 任务备注。

验收标准：

- 用户可以创建多日期、多乘客、多席别任务。
- 用户可以同时运行多个任务。
- 服务重启后可以恢复运行中和查询中的任务。
- 用户可以明确看到每个任务当前状态。
- 任务失败时有明确失败原因。

### 5.5 自动提交普通订单

需求：

- 系统在查询到符合条件余票时自动提交订单。
- 提交订单前必须再次匹配任务条件。
- 提交成功后进入待支付状态。
- 进入待支付状态后，不继续重复提交该任务。
- 系统提醒用户去 12306 支付。

验收标准：

- 有符合条件余票时可以自动提交订单。
- 提交成功后 CLI 和 Web UI 都能看到待支付状态。
- 系统不会自动支付。
- 系统不会在待支付状态重复提交。

### 5.6 候补与强候补

需求：

- 候补必须由用户显式开启。
- 未开启候补时，系统不得提交候补订单。
- 开启强候补后，当没有符合条件余票时，系统可以自动提交候补订单，并暂停继续刷普通余票。
- 强候补优先级高于继续刷票。
- 候补提交成功或失败都需要记录日志。
- 候补兑现成功后进入待支付提醒状态，系统仍不执行支付。
- 候补已提交任务应低频同步官方状态；在接口完成兼容验证前，必须明确提示用户到官方 12306 核对，不得猜测兑现结果。

验收标准：

- 未开启候补的任务只进行余票监控和普通订单提交。
- 开启强候补的任务在无票时可以进入候补提交流程。
- 用户可以看到候补提交结果。

### 5.7 通知

首期通知方式：

- Web UI 状态提示。
- CLI 输出。
- 本机日志。

通知场景：

- 登录成功。
- 登录失败。
- 需要人工验证。
- 查询失败。
- 任务启动。
- 任务暂停。
- 任务取消。
- 命中余票。
- 正在提交订单。
- 订单提交成功，等待支付。
- 候补提交成功。
- 候补提交失败。
- 登录过期。

验收标准：

- 用户可以从 Web UI 和 CLI 得知关键状态。
- 关键事件会写入本机日志。

### 5.8 存储

需求：

- 使用 SQLite。
- 保存应用配置。
- 保存登录会话必要信息。
- 保存乘客配置。
- 保存任务配置。
- 保存任务状态。
- 保存任务日志。
- 保存普通订单结果。
- 保存候补订单结果。
- 保存任务启动时间、时间范围和座位位置偏好。
- 使用 SQLite WAL 和忙等待，支持服务与后台 worker 并发访问。

验收标准：

- 服务重启后仍能查看历史任务。
- 服务重启后配置不丢失。
- 任务日志可追踪。
- 敏感信息不写入普通日志。

### 5.9 Web UI

页面需求：

- 登录与人工验证页。
- 总览仪表盘。
- 车票查询页。
- 创建任务页。
- 任务列表页。
- 任务详情页。
- 设置页。

关键交互：

- 登录。
- 查看登录状态。
- 查询车票。
- 创建任务。
- 开启或关闭候补。
- 启动、暂停、恢复、取消任务。
- 查看待支付提醒。
- 查看候补结果。
- 查看受阻任务并进入人工验证。
- 查看系统设置、安全提示和后续增强开关状态。

验收标准：

- 用户可以只通过 Web UI 完成主要流程。
- 页面状态与后端任务状态一致。
- 表单校验能阻止明显错误配置。
- 原型中的高优先级状态必须有统一口径：普通订单待支付、候补已提交等待兑现、候补兑现成功待支付、等待人工验证。

页面职责：

- 总览仪表盘：展示待支付、人工验证、候补监控、最近命中、关键日志和当前监控任务。
- 车票查询页：完成站点、日期、无座、候补机会等查询条件输入，并展示车次结果。
- 创建任务页：配置多日期、多乘客、席别优先级、无座兜底、车次筛选、候补和强候补策略、查询间隔。
- 任务列表页：展示任务 ID、路线、日期、乘客、状态和主要操作。
- 任务详情页：展示任务配置摘要、状态进度、强候补决策链、实时日志和待支付行动入口。
- 登录与人工验证页：支持账号密码登录，遇到滑块或验证码时引导用户打开 12306 官方验证页面手动完成。
- 系统设置页：配置监听地址、端口、SQLite 路径、默认查询间隔、日志级别，并展示安全与合规须知。

### 5.10 CLI

命令需求：

```text
12306-rs login
12306-rs logout
12306-rs status
12306-rs query --from <出发站> --to <到达站> --date <日期>
12306-rs task create
12306-rs task list
12306-rs task show <task-id>
12306-rs task start <task-id>
12306-rs task pause <task-id>
12306-rs task resume <task-id>
12306-rs task cancel <task-id>
12306-rs serve
```

验收标准：

- 用户可以通过 CLI 完成查询和任务管理。
- CLI 输出包含任务 ID、状态和关键错误。
- CLI 遇到人工验证时提示打开 Web UI。

### 5.11 部署与运行

需求：

- 支持 macOS 本机运行。
- 支持 Linux 本机运行。
- 支持 Docker 运行。
- 支持配置监听地址、端口、SQLite 路径、日志级别。
- 默认只监听本机回环地址；非回环监听必须配置至少 16 位 API token。

验收标准：

- macOS 可以启动服务。
- Linux 可以启动服务。
- Docker 可以启动服务。
- 用户可以访问 Web UI。
- CLI 可以连接本机服务或使用本地配置。

### 5.12 接口可用性验证

需求：

- 默认参考已有 Python 项目的 12306 接口链路，减少前期探索时间。
- 查询、登录、普通下单、候补四类关键链路实现前，需要做最小 smoke test。
- smoke test 只验证接口能返回可识别结构，不做大规模逆向。
- smoke test 失败时，需要记录接口差异并调整对应模块设计。

验收标准：

- 每个关键链路实现前都有最小验证结论。
- 验证失败不会阻塞无关模块开发。
- 接口差异记录在实现任务或开发日志中。

## 6. 产品流程

### 6.1 普通订单流程

1. 用户登录。
2. 用户查询车票。
3. 用户创建抢票任务。
4. 用户启动任务。
5. 系统按配置轮询余票。
6. 系统发现符合条件余票。
7. 系统提交普通订单。
8. 订单提交成功。
9. 系统进入待支付状态。
10. 系统提醒用户去 12306 支付。

### 6.2 强候补流程

1. 用户登录。
2. 用户创建任务。
3. 用户显式开启候补和强候补。
4. 用户启动任务。
5. 系统查询余票。
6. 没有符合条件余票。
7. 系统提交候补订单。
8. 系统记录候补提交结果。
9. 系统提醒用户查看候补状态。

### 6.3 登录验证流程

1. 用户输入账号密码。
2. 系统发起登录。
3. 如果登录成功，进入已登录状态。
4. 如果需要额外验证，Web UI 展示人工验证入口。
5. 用户完成验证。
6. 系统继续确认登录状态。

## 7. 业务规则

- 只有登录状态有效时才能提交订单或候补。
- 未选择乘客时不能创建抢票任务。
- 未选择日期时不能创建抢票任务。
- 未选择席别且未开启无座时不能创建抢票任务。
- 无座只有用户明确允许时才可提交。
- 候补只有用户明确开启时才可提交。
- 强候补开启后，无票时优先提交候补。
- 普通订单提交成功后不自动支付。
- 普通订单提交成功后任务进入待支付状态，不继续重复提交。
- 登录过期时任务应暂停或进入等待登录状态。
- 请求失败应记录日志并按退避策略重试。
- 敏感信息不得写入普通日志。

## 8. 状态定义

任务状态：

- `created`: 已创建。
- `running`: 运行中。
- `paused`: 已暂停。
- `waiting_login`: 等待登录。
- `verification_required`: 等待人工验证。
- `querying`: 查询中。
- `submitting`: 正在提交普通订单。
- `pending_payment`: 订单待支付。
- `candidate_submitting`: 正在提交候补。
- `candidate_submitted`: 候补已提交。
- `candidate_pending_payment`: 候补兑现成功待支付。
- `reconciliation_required`: 提交结果未知，需要先核对官方订单。
- `failed`: 失败。
- `cancelled`: 已取消。

登录状态：

- `logged_out`: 未登录。
- `logging_in`: 登录中。
- `logged_in`: 已登录。
- `verification_required`: 需要人工验证。
- `expired`: 登录已过期。
- `failed`: 登录失败。

## 9. 数据需求

需要持久化的数据：

- 应用配置。
- 登录会话必要信息。
- 乘客信息。
- 抢票任务。
- 任务日期。
- 任务乘客。
- 任务席别。
- 任务车次偏好。
- 任务状态。
- 任务日志。
- 普通订单结果。
- 候补订单结果。
- 任务执行设置和跨进程请求限速记录。

不建议持久化的数据：

- 明文密码。
- 不必要的接口原始响应。
- 会暴露敏感信息的完整请求日志。

## 10. 安全与合规约束

- 不实现自动支付。
- 不实现验证码破解。
- 不实现风控绕过。
- 不鼓励或默认使用高频请求。
- 默认查询间隔应保守。
- 自动打码、自动滑块、代理、CDN 和更激进查询策略不进入 MVP。
- 后续如支持验证辅助能力，必须以人工验证辅助或用户显式配置为边界。
- 后续如支持代理或 CDN，必须由用户显式开启。
- 提供清晰日志，让用户知道系统执行了哪些自动操作。
- 用户需要对自己的 12306 账号和购票行为负责。
