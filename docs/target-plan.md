# 12306-rs 目标计划

## 1. 项目目标

`12306-rs` 是一个使用 Rust 开发的 12306 查询与抢票服务，面向本机和 Docker 部署场景，提供 Web UI 和 CLI 两种入口。

项目要解决的问题是：让用户可以配置出行条件、乘客、席别、车次、日期和候补策略，由系统持续查询余票，并在满足条件时自动提交订单。订单提交成功后停在待支付状态，提醒用户前往 12306 完成支付。

## 2. 已确认需求

- 使用 Rust 语言开发。
- 支持 Web UI。
- 支持终端 CLI。
- 支持本机运行。
- 支持 Docker 服务运行。
- 数据存储使用 SQLite。
- 首期支持 macOS 和 Linux。
- 首期支持账号密码登录。
- 登录遇到额外验证时，通过 Web UI 引导用户人工完成。
- 支持车票查询、余票监控、自动提交订单。
- 自动提交订单后不自动支付，只提醒用户去支付。
- 支持多乘客。
- 支持多日期。
- 支持多任务。
- 支持无座。
- 支持候补和强候补。
- 强候补需要用户在任务中显式开启。
- 当强候补开启且无符合条件余票时，系统可以自动提交候补订单，并且候补优先级高于继续刷票。
- 候补兑现成功后进入待支付提醒状态，仍由用户手动前往 12306 官方渠道支付。
- 默认查询间隔为 3000ms，最小查询间隔为 1000ms。
- 首期通知方式为 Web UI 状态提示、CLI 输出和本机日志。

## 3. 非目标

以下内容不进入首期目标：

- 不做自动支付。
- 不做验证码破解。
- 不做风控绕过。
- 不做绕过 12306 官方限制的高频请求。
- 不做自动打码或自动滑块处理的首期实现。
- 不做代理池或 CDN 加速的首期实现。
- 不做 Windows 首期支持。
- 不做短信、邮件、企业微信、Telegram 等外部通知。
- 不做复杂分布式部署。
- 不做多账号协同抢票。
- 不做浏览器插件。

后续增强可以考虑：

- CDN 查询优化。
- 用户自配置代理。
- 更细的查询间隔策略。
- 验证辅助适配器。
- 外部消息通知。

后续增强必须遵守以下边界：

- 默认查询间隔仍需保守。
- 代理和 CDN 只能作为用户显式配置的优化能力。
- 验证相关能力不得写成验证码破解或风控绕过。
- 后续增强不能影响首期 MVP 的稳定性和清晰边界。

## 4. 成功标准

项目首期成功标准如下：

- 用户可以通过 CLI 登录、查询车票、创建抢票任务、查看任务状态、停止任务。
- 用户可以通过 Web UI 登录、查询车票、配置抢票任务、查看任务状态和日志。
- 系统可以保存用户配置、任务配置、任务状态和关键日志到 SQLite。
- 系统可以同时运行多个抢票任务。
- 单个任务可以支持多个日期、多个乘客、多个席别和多个车次条件。
- 当发现符合条件余票时，系统可以自动提交订单。
- 自动提交订单成功后，系统停止继续抢该订单并提示用户支付。
- 当用户开启强候补且无符合条件余票时，系统可以自动提交候补订单。
- 网络失败、登录过期、接口失败、无票等情况不会导致服务崩溃。
- 系统有请求限速、重试和退避策略。
- 项目可以在 macOS 和 Linux 本机运行。
- 项目可以通过 Docker 运行。

## 5. 推荐技术方向

### 5.1 Rust 后端

建议使用 Rust workspace 管理多个模块：

- `core`: 通用领域模型、筛选逻辑、任务状态机。
- `client_12306`: 12306 HTTP 客户端、登录、查询、订单、候补接口封装。
- `storage`: SQLite 存储、迁移、配置读写。
- `scheduler`: 抢票任务调度、轮询、并发控制、重试退避。
- `server`: Web API 服务。
- `cli`: 命令行入口。
- `web`: Web UI 前端资源。

具体框架可以在实现前根据仓库实际情况确认。默认建议：

- HTTP 服务: Axum。
- 异步运行时: Tokio。
- SQLite: sqlx 或 rusqlite。
- CLI: clap。
- 配置: toml 或 yaml。
- 日志: tracing。
- Web UI: 简洁的 SPA 或服务端渲染页面。

### 5.2 架构原则

- CLI 和 Web UI 复用同一套核心逻辑。
- 12306 接口封装与业务调度解耦。
- 任务调度不直接依赖 UI。
- SQLite 只保存必要数据，不保存不需要落盘的临时状态。
- 登录凭据、Cookie、Token 等敏感信息需要最小化存储，并考虑本机文件权限。
- 自动化操作必须保留清晰日志，方便用户知道系统做了什么。

## 6. 核心模块计划

### 6.1 12306 客户端模块

职责：

- 账号密码登录。
- 识别登录态是否有效。
- 遇到额外验证时返回明确状态，交给 Web UI 引导人工完成。
- 查询站点、车次、余票、席别。
- 提交普通订单。
- 提交候补订单。
- 处理 12306 接口错误、登录过期、会话失效。

验收：

- 能完成一次指定日期、出发站、到达站的余票查询。
- 能识别登录成功、登录失败、需要人工验证三类状态。
- 能在登录态有效时进入订单提交流程。

### 6.2 查询与筛选模块

职责：

- 根据日期、出发站、到达站查询车票。
- 支持车次筛选。
- 支持席别筛选。
- 支持无座。
- 支持多日期查询。
- 根据用户优先级排序结果。

验收：

- 给定任务条件后，可以稳定产出符合条件的候选车次列表。
- 无票、有票、部分席别有票、仅无座有票等状态能被正确区分。

### 6.3 任务调度模块

职责：

- 创建、启动、暂停、恢复、取消任务。
- 支持多任务并发运行。
- 控制查询频率。
- 对失败请求进行重试和退避。
- 维护任务状态。
- 触发普通订单或候补订单提交。

任务状态建议：

- `created`: 已创建。
- `running`: 运行中。
- `paused`: 已暂停。
- `waiting_login`: 等待登录。
- `verification_required`: 等待人工验证。
- `querying`: 查询中。
- `submitting`: 正在提交订单。
- `pending_payment`: 已生成待支付订单。
- `candidate_submitting`: 正在提交候补。
- `candidate_submitted`: 已提交候补。
- `candidate_pending_payment`: 候补兑现成功待支付。
- `failed`: 失败。
- `cancelled`: 已取消。

验收：

- 多任务可以同时运行，互不阻塞。
- 单个任务失败不会影响整个服务。
- 任务状态可以被 CLI 和 Web UI 查询。

### 6.4 订单模块

职责：

- 根据候选车次、乘客和席别提交普通订单。
- 提交成功后记录待支付状态。
- 失败后根据错误类型决定停止、重试或回到查询。
- 不执行支付动作。

验收：

- 有符合条件余票时，可以自动提交订单。
- 提交成功后不继续重复提交同一任务。
- 用户能看到明确的支付提醒。

### 6.5 候补模块

职责：

- 仅当用户显式开启候补时生效。
- 当没有符合条件余票时，按任务条件提交候补订单。
- 强候补开启时，候补优先级高于继续刷票。
- 记录候补提交结果。

验收：

- 未开启候补的任务不会提交候补订单。
- 开启强候补且无票时，会优先尝试候补。
- 候补成功后任务状态可见。

## 7. 抢票流程

普通抢票流程：

1. 用户登录。
2. 用户创建任务。
3. 系统校验任务配置。
4. 系统按任务条件轮询余票。
5. 查询到符合条件车票。
6. 系统进入提交订单状态。
7. 系统提交订单。
8. 提交成功后记录待支付订单。
9. 系统提醒用户去 12306 支付。
10. 任务停止或进入已完成状态。

异常处理：

- 登录过期: 暂停任务并提示重新登录。
- 网络异常: 按退避策略重试。
- 接口限流: 降低请求频率并记录日志。
- 提交失败: 根据错误类型决定重试或停止。
- 订单已存在: 提醒用户处理待支付订单。

## 8. 强候补流程

强候补流程：

1. 用户在任务中开启候补。
2. 系统查询余票。
3. 没有符合条件余票。
4. 系统检查是否满足候补提交条件。
5. 若满足条件，系统提交候补订单。
6. 候补提交成功后记录候补已提交状态。
7. 候补兑现成功后进入候补待支付状态。
8. 系统提醒用户前往 12306 官方渠道手动支付。

约束：

- 未开启候补时不提交候补。
- 强候补优先级高于继续刷票。
- 候补失败需要记录明确原因。

## 10. 风险与控制

### 10.1 12306 接口变化

风险：接口字段、登录流程、订单流程可能变化。

控制：

- 默认先按参考项目接口链路推进，避免前期过度探索。
- 每个关键链路实现前做最小 smoke test，而不是大规模逆向。
- 将 12306 接口封装在独立模块。
- 为解析逻辑增加测试样例。
- 错误日志保留必要上下文。

### 10.2 登录与验证

风险：账号密码登录可能遇到人工验证。

控制：

- 将额外验证作为正常状态处理。
- Web UI 提供人工验证入口。
- CLI 提示用户打开 Web UI 完成验证。

### 10.3 风控和限流

风险：高频请求可能触发限制。

控制：

- 默认使用保守轮询间隔。
- 对失败和限流做退避。
- 不实现绕过限制的逻辑。

### 10.4 自动提交误操作

风险：错误条件导致提交了用户不想要的订单。

控制：

- 创建任务前展示完整确认信息。
- 自动提交前严格匹配乘客、日期、车次、席别。
- 提交成功后立即停止重复提交。

### 10.5 敏感信息

风险：账号、密码、会话信息泄露。

控制：

- 密码不写入普通日志。
- 本机文件权限收紧。
- 后续可增加系统 keychain 或加密存储。

### 10.6 参考项目偏差

风险：参考项目包含旧接口、自动打码、自动滑块、代理、CDN、高频查询、多账号和分布式能力，直接照搬会偏离当前产品边界。

控制：

- 参考项目只作为流程和模型参考。
- 自动打码、自动滑块、代理、CDN、更激进查询策略只进入后续增强。
- 多账号和分布式不进入首期。
- 强候补策略以本项目已确认定义为准：用户开启强候补后，无符合条件余票时，候补优先于继续刷票。
